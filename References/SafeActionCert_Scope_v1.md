# Trier OS Safe Action Certification Layer: Scoping & Architecture

## 1. Constraint Inventory
The certification engine must systematically check preconditions before authorizing safety-critical writes. The inventory of constraints includes:

- **LOTO Constraint:** *Can this asset be safely worked on?* 
  - **Check:** Ensure no open LOTO permits with unverified isolation points cover the target asset.
  - **Data:** Query `loto_permits` WHERE `AssetID = ?` AND `Status NOT IN ('CLOSED', 'VOID')`.
  - **Protected Routes:** Work Order Close (`POST /api/jobs/:id/close`), Asset Energize.
- **Permit to Work (PTW) Constraint:** *Is the required safety permit active?*
  - **Check:** Operations like hot work or confined space entry require an ACTIVE permit and cleared gas readings.
  - **Data:** Query `safety_permits` for matching type and active status.
  - **Protected Routes:** Work Order Close, Permit Sign-Off.
- **MOC Constraint:** *Does this configuration change have an approved MOC?*
  - **Check:** Ensure the `moc` table contains an `APPROVED` record referencing the affected asset/system before allowing structural asset modifications.
  - **Data:** Query `moc` table.
  - **Protected Routes:** Asset Modification, Work Order Close.
- **Safety Incident Constraint:** *Is there an open high-severity incident on this asset?*
  - **Check:** Block maintenance work on the asset until the incident is formally escalated or resolved.
  - **Data:** Query `AuditLog` or incidents table for open critical events.
  - **Protected Routes:** Work Order Creation/Close.
- **System State Constraint:** *Is the system operating in NORMAL mode?*
  - **Check:** Pre-validated globally by `degradedMode.js` (blocks non-NORMAL writes).
- **Criticality Constraint:** *Does this action require dual-authorization?*
  - **Check:** If the asset is Criticality 'A', block single-party closure; require second-party sign-off.
  - **Data:** Query `Asset` table `CriticalityRating`.
  - **Protected Routes:** Work Order Close, LOTO Clear.

## 2. Proof Format (Certified Safe Receipt)
When all constraints pass, the system generates a cryptographically verifiable receipt.
- **Schema:** 
  - `ReceiptID`: UUID
  - `ActionType`: String (e.g., 'WORK_ORDER_CLOSE')
  - `TargetID`: Target entity ID
  - `Timestamp`: ISO-8601 UTC
  - `ConstraintsEvaluated`: Array of `{ name, result }`
  - `SystemState`: The global operating mode at the time of certification
  - `Expiry`: Timestamp (TTL set to 15 minutes to prevent stale authorization)
- **Signature:** HMAC-SHA256 hash using the server's `JWT_SECRET`. This guarantees the receipt was generated by the trusted Trier OS instance and has not been tampered with.
- **Storage:** Dedicated `CertificationReceipts` table.
- **Consumption:** The client calls a `POST /api/certify` endpoint first, receives the receipt, and embeds the `ReceiptID` in the subsequent write payload.

## 3. Causal Explanation for Blocked Actions
When an action is rejected, the system must return a structured explanation rather than a generic 403 error.
- **Response Format:**
  ```json
  [
    {
      "constraint": "LOTO_OPEN",
      "blockingEntities": [{"id": "LOTO-102", "holder": "John Doe", "details": "Unverified isolation points"}],
      "remediation": "Verify all isolation points or contact John Doe."
    }
  ]
  ```
- **Frontend Integration:** Operators are presented with a detailed, actionable modal ("Cannot proceed — 1 issue must be resolved") containing direct hyperlinks to the blocking entities (e.g., clicking 'LOTO-102' opens the specific permit).

## 4. Simulation Engine Assessment
The Parallel Universe module in `live_studio.js` utilizes `fs.copyFileSync` to clone a plant database (potentially 50-200MB) for KPI simulations. 
- **Assessment:** Synchronously cloning the database for every routine safety certification request is prohibitively slow and resource-intensive.
- **Recommendation:** Standard pre-write certification must utilize optimized, read-only constraint queries against the live database without cloning. The Parallel Universe cloning pattern should remain strictly reserved for long-running, macroscopic "what-if" planning sessions.

## 5. Gatekeeper Dependency
P2-2 Gatekeeper (Command Dispatcher) remains partially deferred.
- **Path A (Wait for Gatekeeper):** Comprehensive system-wide certification. Every database write is intercepted by a unified gatekeeper that enforces the checks.
- **Path B (Targeted v1):** Manually bolting certification checks onto 4-5 high-risk routes immediately, bypassing the need for a global gatekeeper.
- **Risk Assessment:** Path B creates a severe "illusion of safety." In industrial operations, partial safety mechanisms are actively dangerous. If operators are conditioned to see a "Certified Safe" badge on some actions, they will implicitly assume the absence of a warning on an uncovered route means it is equally safe. A false negative here invites catastrophic operator error.

## 6. Decision
**Decision: NO-GO for both full and targeted implementation.**

- **Full system-wide certification:** NO-GO. Blocked pending the P2-2 Gatekeeper implementation.
- **Targeted v1 on highest-risk routes:** NO-GO. Explicitly rejected. Implementing partial certification breeds operator complacency and introduces unacceptable safety risks due to false confidence on uncertified routes.

**Blocking Prerequisite:** The P2-2 Gatekeeper must be implemented. A centralized write-intercept architecture is non-negotiable to ensure 100% constraint coverage across all system mutations.
