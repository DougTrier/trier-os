**Created:** April 24, 2026
**Source:** Roadmap Parts and Assets.md / Write_Path_Architecture.md / RBAC_and_Action_Model.md
**Version at creation:** v3.5.1
**Prerequisite state:** P7-3-impl and P7-4-impl complete. P7-1 (Time Machine) and P7-2 (Safe Action Cert) are blocked NO-GO pending this task list.

---

## Why This Task List Exists

Trier OS currently executes all database mutations via raw SQLite queries scattered across Express route handlers. There is no centralized enforcement layer. This means:
- Safety-critical actions (LOTO, MOC, OPC-UA writes) have no automated validation gate
- P7-1 (Time Machine) cannot capture atomic before/after payloads with guaranteed completeness
- P7-2 (Safe Action Cert) cannot pre-validate intents against plant state
- A plant safety officer cannot certify a system where any write bypasses enforcement

This task list builds the Gatekeeper Service: a separate runtime that sits between Trier OS and all governed write paths. When this task list is complete, those two P7 features become implementable.

**Important scope boundary:** This task list connects the Gatekeeper to the known safety-critical paths (LOTO, MOC, OPC-UA, Modbus). The full "intercept every write" requirement -- needed to fully unblock P7-1 and P7-2 -- is the next increment. Document it as such in each completion gate.

## Task G1: Gatekeeper Service Foundation

> Establishes the Gatekeeper as an isolated Node.js process with a defined IPC contract. Nothing else in this task list is buildable until G1 is complete.

**Before starting:**
- [x] Re-read CLAUDE.md
- [x] Re-read server/standards.md in full

**IPC Decision (record here before starting sub-tasks):**
The Gatekeeper runs as a standalone Express process on `127.0.0.1` port `4001` (configurable via `GATEKEEPER_PORT` env var). Trier OS communicates with it via HTTP POST on the loopback interface. This is chosen over Unix sockets (Windows incompatible) and named pipes (complex cross-platform). The loopback-only binding ensures the Gatekeeper is never reachable from outside the host machine.

**Sub-tasks:**

- [x] Create directory `server/gatekeeper/` to house all Gatekeeper modules
- [x] Create `server/gatekeeper/index.js`: standalone Express server
  - Binds to `127.0.0.1:${GATEKEEPER_PORT || 4001}` only -- never `0.0.0.0`
  - Exposes one route: `POST /validate-intent`
  - Loads and calls the Change Validation Engine (stub for now, filled in by G6)
  - Returns `{ allowed: true|false, requestId, denialReason, auditRef }` 
  - On any internal error: returns `{ allowed: false, denialReason: 'GATEKEEPER_INTERNAL_ERROR' }` -- never propagates an unhandled exception as an implicit allow
  - Copyright header required as first line
- [x] Create `server/gatekeeper/client.js`: Trier OS-side HTTP client
  - `validateIntent(intent)`: sends POST to Gatekeeper, returns the parsed response
  - **Fail-closed contract:** if `GATEKEEPER_URL` is not set, returns `{ allowed: false, denialReason: 'GATEKEEPER_NOT_CONFIGURED' }`
  - **Fail-closed contract:** if Gatekeeper is unreachable (ECONNREFUSED, timeout, non-200), returns `{ allowed: false, denialReason: 'GATEKEEPER_UNREACHABLE' }` -- NEVER throws or returns allowed:true on network failure
  - Timeout: 5000ms. Use Node.js built-in `http` module -- no axios or node-fetch
  - Copyright header required
- [x] Define the Intent schema (document in `server/gatekeeper/schema.js`):
  ```js
  {
    requestId:   string,   // UUID v4, generated by caller (S-12 collision-resistant)
    userId:      string,   // req.user.UserID (from JWT)
    username:    string,   // req.user.Username (capital U)
    plantId:     string,   // validated against SAFE_PLANT_ID
    actionClass: string,   // READ_ONLY | ADVISORY | NON_CRITICAL | SAFETY_CRITICAL
    actionType:  string,   // semantic label e.g. 'LOTO_ACTIVATE', 'MOC_APPROVE', 'SETPOINT_WRITE'
    targetType:  string,   // ASSET | WORK_ORDER | LOTO_PERMIT | MOC_RECORD | FIELD_DEVICE
    targetId:    string,   // ID of the specific entity being acted on
    payload:     object,   // the proposed write (before snapshot)

  }
  ```

  - Columns: `LedgerID INTEGER PRIMARY KEY`, `RequestID TEXT NOT NULL UNIQUE`, `Timestamp TEXT NOT NULL`, `UserID TEXT NOT NULL`, `Username TEXT NOT NULL`, `PlantID TEXT`, `ActionClass TEXT NOT NULL`, `ActionType TEXT NOT NULL`, `TargetType TEXT`, `TargetID TEXT`, `ValidationResult TEXT NOT NULL` (ALLOWED | DENIED), `DenialReason TEXT`, `PTWRef TEXT`, `MOCRef TEXT`, `RoleAtDecision TEXT`, `ProcessingMs INTEGER`
  - Index on `(PlantID, Timestamp DESC)`
  - Index on `(UserID, Timestamp DESC)`

  - Standard migration format: `module.exports = { up: () => { const logisticsDb = require('../logistics_db').db; ... } }`

## Task G6: Change Validation Engine

> The orchestration core of the Gatekeeper. Runs all validators in sequence and produces a single Allow/Deny decision. G3, G4, and G5 must all be complete before starting.

**Before starting:**
- [ ] Re-read CLAUDE.md
- [ ] Re-read server/standards.md
- [ ] Confirm Tasks G3, G4, and G5 are all fully marked complete

**Validation pipeline (in order -- stop at first failure, do not short-circuit early passes):**

```
1. Identity Check   — Is the JWT valid? Does req.user exist?
2. Action Registry  — Is the actionType registered? What class is it?
3. RBAC Check       — Does the user's AD role permit this actionClass? (G3)
4. State Check      — Is the target asset currently in LOTO? (Block NON_CRITICAL if LOTO active)
5. PTW Check        — (SAFETY_CRITICAL only) Is there an active PTW covering this user+asset? (G4)
6. MOC Check        — (SAFETY_CRITICAL + MOC-required action types only) Is there an approved MOC? (G5)
7. Audit            — Write to GatekeeperAuditLedger regardless of Allow/Deny
8. Decision         — Return { allowed, requestId, denialReason, auditRef }
```

**Sub-tasks:**

- [x] Create `server/gatekeeper/engine.js`
  - `validateIntent(intent)`: runs the full pipeline above
  - Step 4 (State Check): query `LotoPermits` for any ACTIVE permit on `intent.targetId` and `intent.plantId`. If found and the requesting user is NOT the ISSUER or WORKER on that permit, block NON_CRITICAL writes too -- an asset under LOTO is isolated from all writes except by the permit holder
  - The engine must measure its own processing time (`Date.now()` before and after) and include `processingMs` in the audit record
  - On any validator throwing an exception: catch it, log to stderr, and return `{ allowed: false, denialReason: 'VALIDATOR_EXCEPTION' }` -- never propagate
- [x] Wire `engine.js` into `server/gatekeeper/index.js` (replacing the stub from G1)
- [x] Update the `POST /validate-intent` response to include all fields from the engine output
- [x] Add `processing_time_ms` logging to every Gatekeeper decision (helps identify slow AD lookups)

**State Check detail:** The state check is distinct from the PTW check. PTW asks "do you have authorization to work on this isolated asset?" State check asks "is this asset currently isolated at all?" A NON_CRITICAL write to an asset under LOTO should be blocked even if it doesn't require a PTW normally -- someone may be working inside the machine.

**Completion gate:**
- [x] Full validation pipeline runs in correct order
- [x] All six validation steps produce correct Allow/Deny in integration tests
- [x] Every decision logged to `GatekeeperAuditLedger` with full chain (PTWRef, MOCRef, RoleAtDecision)
- [x] Engine catches and contains all validator exceptions (never allows on exception)
- [x] Reviewed against standards -- no violations
